Anatomy of a Skill

A skill is a folder that teaches Claude a repeatable capability. The only required file is SKILL.md: YAML frontmatter plus a Markdown body. Claude loads a skill's one-line description at all times, and pulls in the rest only when it is needed.

The shape

A skill is a directory. The folder name becomes the skill's name and its /command. Everything except SKILL.md is optional supporting material the skill can reach for.

  • RRequired: exactly one SKILL.md at the folder root.
  • +Optional: reference docs, scripts, templates, sample data. The body decides when to use them.
.claude/skills/
└── summarise-changes/     the skill (folder = name)
    ├── SKILL.md           required: metadata + body
    ├── reference.md       optional: detail, on demand
    └── scripts/
        └── helper.sh      optional: code it can run

Anatomy of SKILL.md

---                                    1
name: summarise-changes                2
description: Summarise uncommitted      3
  changes; use when asked what changed.
allowed-tools: Bash(git diff:*)        4
---

# Summarise changes                    5

## Workflow
1. Run `git status` and `git diff`.
2. Group the changes by file and intent.
3. Summarise; draft a commit if asked.

See `reference.md` for the style guide. 6
  • 1Frontmatter fence. YAML metadata between --- markers. Its name and description are the part always kept in context.
  • 2name: the skill's identifier and its /command name. Optional; the folder name is the fallback.
  • 3description, the one that matters: what it does and when to use it. Claude matches your request against this to decide whether to invoke the skill.
  • 4allowed-tools: optional. Tools pre-approved while the skill runs, so it is not stopped for permission mid-task.
  • 5Body. Plain Markdown: the runbook (workflow, commands, notes, examples). Loaded when the skill activates.
  • 6Pointer to a supporting file. Loaded only when Claude actually follows it. This is progressive disclosure at work.

Frontmatter fields

nameIdentifier and /command name. Folder name is the fallback.
descriptionCritical. What it does + when to use it. The trigger for auto-invocation.
allowed-toolsTools pre-approved while active, e.g. Bash(git diff:*).
argument-hintAutocomplete hint, e.g. [issue-number].
moreOthers exist (model, effort, paths, context). See the docs.

Progressive disclosure

Always loaded
Every skill's name + description. A few lines each, so Claude knows what exists.
On activation
The full SKILL.md body, when the skill triggers or you invoke it.
On demand
Supporting files, only when the body points Claude to them.
Keep SKILL.md lean; push detail into supporting files that cost nothing until read.

Discovery & invocation

Automatic
Claude reads every skill's description and invokes the one that fits your request.
Manual
Type /skill-name (with any arguments) to run it directly.
Skills vs commands
Same format. A skill is a command that lives in its own folder, can bundle files, and can auto-invoke. .claude/commands/x.md and .claude/skills/x/SKILL.md both give /x.

Where skills live

Personal
~/.claude/skills/<name>/SKILL.md. Available in all your projects.
Project
.claude/skills/<name>/SKILL.md. Committed, shipped with one repo.
Plugin
Bundled in an installed plugin, namespaced plugin:skill.

Write the description

It is the only part always in context, so it has to earn the invocation.
Say what it does and when: "Search the local Dyalog docs. Use when asked about a symbol or system function."
Include the words a user would actually say. Vague ("handles code tasks") never fires.

Gotchas

Vague description: Claude never auto-invokes it.
Oversized SKILL.md: keep it lean, point to supporting files.
Edited a skill mid-session? Run /reload-skills to re-scan.
A file in .claude/commands/ works but misses the folder and supporting-files features. Use .claude/skills/<name>/SKILL.md for the full set.
