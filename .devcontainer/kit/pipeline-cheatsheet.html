Dyalog · Claude Code · working pipeline

How we work

Idea to merged PR, one lane. Work from reviewed, versioned artefacts, not throwaway prompts. Every step does one thing, then stops for a human to look before the work moves on.

Plan & issues Tests (RED) Build (GREEN) Ship Bug work
1

Idea to plan

Plan mode

On a clean main, enter plan mode (shift-tab) and describe the goal. Claude explores read-only and drafts a plan. Read it closely and push back before you accept it.

TypePlan <what you want to build>. It should…
Review/dyalog:crev docs/plans/<slug>.md (separate window, optional but wise)
SaveSave the plan to docs/plans/<slug>.md and remove the temporary file
Makesdocs/plans/<slug>.md
2

Plan to issues

GitHub

Turn the plan into trackable work. One Epic groups the child issues; each issue carries its own acceptance criteria and points back at the plan.

TypeConvert docs/plans/<slug>.md to GitHub Epics and linked sub-issues, each referencing the plan document for context.
Makes1 Epic + child issues with acceptance criteria
CheckOpen them on GitHub. The rendered version often reads differently from the source.
3

Write the tests

RED

One prompt per step. Claude creates the branch, writes tests that fail for the right reason (one behaviour each, no shared state), records them, and stops. Nothing is committed until the review approves it.

TypeProceed <issue-number>
DoesBranch <id>-<slug> · failing tests · writes the PR doc · stops.
Review/clear then /dyalog:crev <id>
LoopApproved; proceed  or  Read the review at docs/reviews/<id>.md and address its findings  until approved.
Makesdocs/prs/<id>.md · docs/reviews/<id>.md
4

Make them pass

GREEN

Same one-step rhythm. On approval Claude commits the tests, writes just enough code to turn them green, runs the full suite and linters, updates the PR doc, and stops for the second review.

TriggerApproved; proceed
DoesCommits approved tests · implements to green · full suite + linters · updates PR doc · stops.
ReviewSame as RED, but keep the context this time. No /clear.
5

Open the PR, then merge

Ship

When the issue is done, push and open the PR. You merge it after human review and green CI. The pipeline never merges for you.

TypeRun the full test suite. If it passes, push the branch and open a pull request that summarises the cycles in docs/prs/<id>.md.
MergedPR merged. Pull from upstream and remove my local branch which is now merged.

Bug work — alternate start

Bugs enter through their own command, then join the cycle at step 3.

Start/dyalog:bugfix <issue-number>

Reproduces the bug for real, writes failing regression tests, and writes docs/bugs/<id>.md: verified repro, root cause (file, function, why), fix outline, and an Also Impacts section. No production code.

If it cannot reproduce, it stops. Do not let it proceed to a fix.

Then /dyalog:crev <id>, and continue from step 3.

Reading a review

CriticalMust fix before merge. Correctness, regressions.
MajorShould fix before merge. Gaps, API issues.
MinorNice to fix. Style, docs, small wins.
NoteNo change needed. Debt, follow-up ideas.
Verdict: Approve · Approve with minor changes · Request changes

A Critical means Request changes. Minor findings and Notes never gate the verdict.

Two windows

Window 1, planner / implementer. Drives the cycle with Proceed.

Window 2, reviewer. Runs /dyalog:crev in a fresh context.

/clear the reviewer before the RED (tests) review. Keep its context for the GREEN (implementation) review.

Where things live

  • docs/plans/Feature plans
  • docs/bugs/Repro, RCA, fix outline
  • docs/prs/PR notes, append-only, one entry / cycle
  • docs/reviews/Review log per issue

Done when

  1. The full test suite passes.
  2. Linters and type checks pass.
  3. The PR doc is honest across all cycles.
  4. Latest verdict is Approve (or Approve with minor changes).

Guardrails (hooks enforce)

  • No --force push · no push to main
  • No --no-verify · no bulk git add · no commit -a
  • No rm -rf of / ~ .git · no curl | sh · no sudo
  • Can't touch .claude/ hooks, settings, .mcp.json
  • Secrets (.env, keys) are unreadable

When to skip the pipeline

Trivial fix: under 10 lines, no behaviour change. Edit and commit directly. Hooks still apply.
Urgent prod fix: fix, ship, post-mortem after. Point bugfix at the pre-fix commit.
Exploratory spike: throwaway branch, outside the pipeline. Discard it, then plan properly.
