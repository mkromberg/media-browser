# PR notes: Issue #15 - ParseEXIF core (JPEG/TIFF binary parser)

Plan: https://github.com/mkromberg/media-browser/blob/main/docs/plans/original-plan.md#L200-L214

Part of epic #5 (EXIF metadata).

This branch is an Opus redo of the same issue for model comparison. It is developed independently of the existing PR 29 work and does not read that implementation.

## Scope

`meta←ParseEXIF path`: a native binary JPEG/TIFF parser with no `exiftool`, no `ffprobe`, no external image library. It reads a bounded prefix of the file, confirms the JPEG SOI, walks the segment markers for an APP1 segment whose payload begins `Exif\0\0`, parses the embedded TIFF header and IFD0, follows the Exif (`0x8769`) and GPS (`0x8825`) sub-IFD pointers, and resolves the tag set `Make`, `Model`, `Orientation`, `DateTimeOriginal`, `ExifImageWidth`, `ExifImageHeight`, `GPSLatitudeRef`/`GPSLatitude`, `GPSLongitudeRef`/`GPSLongitude`.

The result is always a namespace with `.Found` plus every tag field, present whether or not EXIF was found. An absent or unresolvable field is uniformly `''`/`⍬`, never `⎕NULL`, and no input raises. GPS latitude and longitude are returned as signed decimal degrees (south and west negative).

Integration into `BuildFileMeta` (nesting the result under `.exif` for `type='image'`) is issue #16, out of scope here.

## Fixtures

`apl-service/tests/fixtures/exif/generate_fixtures.apls` hand-builds seven binary fixtures byte by byte (JPEG markers, TIFF header, IFD entries, RATIONALs) with no third-party image library, matching the project's dependency-free constraint. It runs under `⎕IO←0` so byte offsets index the arrays directly, states every TIFF-relative offset as a named constant, and concatenates the blocks in that same order so the layout is verified by construction. It ends with an independent self-check that reads `with_exif.jpg` back from disk and decodes the `Make` string at its known offset without `ParseEXIF`.

The issue text names four fixtures (`with_exif.jpg`, `no_exif.jpg`, `not_an_image.png`, `corrupt_truncated.jpg`) but the acceptance criteria lists seven distinct degradation cases. Three more fixtures were added, each isolating exactly one of the remaining cases:

- `malformed_tiff_header.jpg`: the TIFF byte-order marker corrupted to `XY` (neither `II` nor `MM`), everything else intact.
- `out_of_bounds_ifd0.jpg`: the `Make` entry's value/offset field repointed past EOF, every other entry intact.
- `zero_denominator_gps.jpg`: the `GPSLatitude` first RATIONAL's denominator zeroed, everything else intact.

Flagging this deviation from the literal fixture list for review.

The well-formed fixture encodes `Make='Dyalog'`, `Model='TestCam'`, `Orientation=1`, `DateTimeOriginal='2026:07:10 12:34:56'`, `ExifImageWidth=800`, `ExifImageHeight=600`, and GPS `51 deg 30 min 0 sec N` / `0 deg 7 min 30 sec W`, which resolve to `GPSLatitude=51.5` and `GPSLongitude=-0.125`.

### Anticipated implementation

- Bounded read: a single APP1 segment cannot exceed 65535 bytes (its length field is 16-bit), so reading a ~70000-byte prefix covers SOI plus any one APP1 segment without loading whole files.
- Marker walk: confirm SOI, step over segments by their big-endian length (segment lengths are big-endian regardless of the TIFF byte order declared inside APP1), stop at SOS/EOF, and identify APP1 by the `Exif\0\0` payload prefix.
- TIFF header: validate `II`/`MM` and the 42 magic, decode the IFD0 offset.
- IFD walk: read IFD0's 12-byte entries, bounds-checked against the actual bytes present; resolve inline (type-size times count no more than 4 bytes) versus offset-referenced values; follow the Exif and GPS sub-IFD pointers. `.Found` is set once IFD0's own table is within bounds, so a single broken entry offset leaves only that field empty.
- GPS: convert each axis's three RATIONAL degree/minute/second values to decimal, apply the hemisphere ref sign, and emit `GPSLatitude`/`GPSLongitude` only if both axes resolve and no RATIONAL has a zero denominator.
- An outer `:Trap 0` is a last line of defence, but each degradation is detected at a specific named step rather than relying on the catch-all.

## Cycle 2: GREEN

### Implementation

`apl-service/APLSource/ParseEXIF.aplf` is a single tradfn running under `⎕IO←0` so byte offsets index the buffers directly. The result namespace is fully initialised up front (every field empty, `.Found←0`), so the contract "every field present, never `⎕NULL`, never raised" holds no matter where parsing stops.

The stages the plan names are inline tradfn control flow, not separate functions: bundling a heterogeneous result such as a byte order paired with a numeric offset through a returned strand risks silently nesting the vector, so the marker walk and TIFF-header validation are `:While`/`:If` blocks assigning `tiff`, `le` and `ifd0Off` in the enclosing scope. Only small single-return-type decoders are factored into local dfns that close over `tiff`/`le`/`nt` rather than taking them as arguments: `DecInt` (byte-order-aware integer decode), `TypeSize`, the four entry-field accessors `Etag`/`Etype`/`Ecount`/`Evalfield`, `SafeU16`/`IFDFits` (bounds checks), `DecASCII`, `ResolveEntry` (inline-versus-offset value resolution) and `ResolveRationals` (GPS DMS to decimal).

The APP1 marker walk clamps a segment's declared length to the bytes actually read, so a truncated file still yields whatever TIFF bytes exist; `IFDFits` (the declared entry table against `≢tiff`) is what turns a truncation into `Found=0`, keeping that degradation at the specific named step rather than at the catch-all. `.Found` is set to 1 the moment IFD0's own table passes `IFDFits`, so a broken value offset inside one entry (`out_of_bounds_ifd0.jpg`) leaves only that field empty rather than clearing `.Found`.

GPS conversion returns `(ok decimal)` from a single dfn: a zero denominator or an out-of-bounds RATIONAL yields `ok=0`. The "populated only if both axes resolve" rule is one guard (`latOk∧lonOk`) ahead of assignment, and the hemisphere sign (`1-2×'S'=⊃latRef`) is applied there, independently of the refs themselves, which are assigned as each is read.

### APL specifics worth recording

- A dfn defined inside a tradfn can read that tradfn's local variables, so the byte decoders close over `tiff`/`le`/`nt` instead of threading them through every call. Verified before relying on it.
- `⊃¨` over the result of `m.⎕NC nameList` compares against the wrong thing: `⎕NC` given a vector of names returns the extended form (`2.1` for a variable, not `2`), so the presence check floors it (`2=⌊m.⎕NC Fields`). This was a defect in the test's assertion mechanism, not the parser; the parser was correct from the first run.
- A single-character `⎕UCS` result is a length-1 vector, which fails to `≡`-match a scalar character literal such as `'N'`. `DecASCII` discloses a length-1 result to a scalar so a one-letter GPS ref compares cleanly.
- Multi-term `+`/`-` and mixed arithmetic/comparison expressions are fully parenthesised: APL evaluates strictly right to left, so `pos+1≤n` would parse as `pos+(1≤n)`.

### How it was tested
