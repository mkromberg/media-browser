# PR notes: Issue #15 - ParseEXIF core (JPEG/TIFF binary parser)

Plan: https://github.com/mkromberg/media-browser/blob/main/docs/plans/original-plan.md#L200-L214

Part of epic #5 (EXIF metadata).

## Scope

`meta←ParseEXIF path`: a native binary JPEG/TIFF parser, no `exiftool`/`ffprobe`, no external image library. Reads a bounded prefix of the file, walks JPEG markers for an APP1 segment holding `Exif\0\0`, parses the TIFF header and IFD0, follows the Exif (`0x8769`) and GPS (`0x8825`) sub-IFD pointers, and resolves the tag set `Make`, `Model`, `Orientation`, `DateTimeOriginal`, `ExifImageWidth`, `ExifImageHeight`, `GPSLatitudeRef`/`GPSLatitude`, `GPSLongitudeRef`/`GPSLongitude`.

The result is always a namespace with `.Found` plus every tag field, present whether or not EXIF was found; absent is uniformly `''`/`⍬`, never `⎕NULL`, never a raised error. Every documented degradation case (missing path, non-JPEG, no APP1, truncated segment, malformed TIFF header, an out-of-bounds IFD entry, a zero-denominator RATIONAL) degrades gracefully rather than erroring.

Integration into `BuildFileMeta` (nesting the result under `.exif` for `type='image'`) is issue #16, out of scope here.

## Fixtures

`apl-service/tests/fixtures/exif/generate_fixtures.apls` hand-builds seven binary fixtures byte-by-byte (JPEG markers, TIFF header, IFD entries, rationals) with no third-party image library, matching the project's dependency-free constraint. It is checked in alongside its output so fixtures can be regenerated and diffed.

The issue text names four fixtures (`with_exif.jpg`, `no_exif.jpg`, `not_an_image.png`, `corrupt_truncated.jpg`) but the acceptance criteria lists seven distinct degradation cases. Three more fixtures were added, each isolating exactly one of the remaining cases: `malformed_tiff_header.jpg` (byte-order marker corrupted to neither `II` nor `MM`), `out_of_bounds_ifd0.jpg` (one IFD0 entry's value/offset field corrupted to point past EOF, everything else intact), `zero_denominator_gps.jpg` (GPS latitude's degrees rational has denominator 0, everything else intact). Flagging this deviation from the literal fixture list for review.

Every fixture was verified independently of the (not yet written) `ParseEXIF`: `generate_fixtures.apls` ends with a self-contained decode of `with_exif.jpg` read back from disk, and a separate throwaway probe (not checked in) decoded all seven files field-by-field to confirm each corruption lands exactly where intended and nowhere else.

## Two APL pitfalls hit while building the fixtures, worth recording

**Mixed scalar/vector strand notation silently nests.** `Entry(271 2 6(4 EncN 74))` bundles three scalars and a 4-byte vector into one argument via bare juxtaposition; APL nests the vector item to keep the outer array's shape uniform, so the "flat" byte array is actually boxed one level deep and downstream `⎕NAPPEND`/indexing misbehaves without a visible error until the byte counts stop matching. Fixed by making `Entry`/`Rational` dyadic (`(tag type count) Entry valueBytes`, `numerator Rational denominator`) so no strand ever mixes a scalar tuple with a vector.

**A bare unassigned non-last statement in a dfn ends the function immediately**, returning that statement's value; it does not get silently discarded in favour of a later line. `WriteBytes`'s `⎕NUNTIE tie` as a bare final-but-one statement meant the function returned early with `⎕NUNTIE`'s (shy) tie-number result and never reached the intended return value in earlier drafts. Fixed by assigning every intermediate statement's result, even when unused (`written←(ToSigned ⍵)⎕NAPPEND tie 83`).

Both are recorded as comments at the point of use in `generate_fixtures.apls` rather than only here, since a future maintainer editing that file is exactly who needs the warning.

### Anticipated implementation

- `FindAPP1`: bounded read (JPEG's 16-bit segment-length field caps a single APP1 segment at 65535 bytes), confirm SOI, walk markers for APP1 with an `Exif\0\0` payload, stop at SOS/EOF with none found. JPEG segment-length fields are always big-endian regardless of the TIFF byte order declared inside APP1.
- `ParseTIFFHeader`: validate `II`/`MM` + 42 magic, decode the IFD0 offset.
- `ReadIFDEntries`/`ResolveEntryValue`: walk IFD0's 12-byte entries (bounds-checked against the file's actual length), resolve inline (≤4 byte) vs offset-referenced values, follow the Exif and GPS sub-IFD pointers.
- GPS decimal degrees: convert each axis's three RATIONAL degree/minute/second values, but only emit `GPSLatitude`/`GPSLongitude` if both axes resolve and neither RATIONAL has a zero denominator.
- Outer `:Trap 0` as a last line of defence, but each degradation case is detected at a specific named step per the plan, not solely by the catch-all.
