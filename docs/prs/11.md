# PR notes: Issue #11 - ResolveSubfolder path-traversal gate

Plan: https://github.com/mkromberg/media-browser/blob/main/docs/plans/original-plan.md#L129

Part of epic #3.

## Scope

`root ResolveSubfolder relFolder` is the only function permitted to turn client-supplied path text into a filesystem path passed to `CheckFolder`/`ScanFolder`/`ServeFile`. Empty `relFolder` returns `root` unchanged. Otherwise it joins `root` and `relFolder`, canonicalises (resolving `.`/`..` and symlinks), and checks the result is either exactly `root` or starts with `root` followed by `/`. Anything else - `../../etc`-style traversal, an absolute-path input, a symlink inside root pointing outside it, or a sibling directory sharing root's name as a bare string prefix - returns `'outside-root'` rather than a path.

## Design decisions and empirical findings

**`⎕NPARTS` does not canonicalise; `realpath(3)` does.** Issue #9's work already established that `⎕NPARTS` leaves a literal `..` segment in place rather than resolving it. Rejecting a symlink that escapes root requires genuine OS-level path resolution (following symlinks, not just collapsing `.`/`..` textually), which nothing in `⎕NPARTS`/`⎕NINFO` provides directly. `realpath` is bound via `⎕NA` against the bare SONAME `libc.so.6` (resolved through the system's normal dynamic-linker search path, no full path or `)COPY` needed - confirmed empirically, simpler than the `quadna` workspace copy technique the Dyalog docs describe) rather than shelling out to a `realpath`/`readlink` binary, keeping the service dependency-free per the project's existing EXIF/vendoring rationale.

**`realpath` requires every path component to exist**, including a component reached only via a trailing `..`. Confirmed empirically: `realpath('.../APLSource/../config/app.config.json')` fails outright if `APLSource` is not a real directory, even though the final target exists and even though naive lexical `..`-collapsing would "resolve" it fine (this was also the exact cause of a fixture bug in issue #10's `InitializeTests`). For `ResolveSubfolder`, this matters because an ordinary request for a subfolder that legitimately does not exist yet (a `CheckFolder` `'not-found'` case, not a traversal attempt) must still produce a within-root path rather than being misclassified as `'outside-root'`. The implementation backs off one path segment at a time from the full joined path until it finds the deepest prefix that actually resolves, checks containment only on that resolved (existing) prefix, and re-appends whatever nonexistent trailing segments were backed off, untouched - a component that does not exist cannot be a symlink, so this is safe.

**An absolute-looking `relFolder` is not silently relativised.** The join step uses ordinary path-join semantics: if `relFolder` starts with `/`, it replaces `root` entirely for the canonicalisation step (matching how most languages' path-join functions treat an absolute second argument), rather than being concatenated onto `root` as text. This is what makes the acceptance criterion's "absolute-path inputs... return `'outside-root'`" hold: the canonicalised absolute path almost never starts with `root,'/'`, so it fails containment directly, rather than being reinterpreted as a same-named relative subfolder under root.

**The sibling-prefix trap needs the trailing separator in the containment check, not just a bare prefix match.** `root,'/'` (not bare `root`) is the required prefix, so a resolved path of `root,'-evil'` (a real sibling directory sharing root's name as a string prefix) correctly fails containment despite `⍷`-style bare prefix matching wrongly admitting it - exactly the case the plan calls out by name.

## Cycle 1: RED

### Test surface

`apl-service/tests/unit/ResolveSubfolderTests.aplf` (8 tests) drives `ResolveSubfolder` against a committed fixture tree under `apl-service/tests/fixtures/resolve-subfolder/`: `root/` (with a real child `sub/`, `sub/nested/`, and `evil-link` - a committed relative symlink to `../outside`), a sibling `root-evil/` sharing root's name as a string prefix, and `outside/`. These are committed as real filesystem entities (including a real symlink) rather than built at test-time, since `realpath` needs every existing component to actually exist on disk to prove the traversal and symlink cases are genuinely rejected, not just textually pattern-matched.

An initial version of the test file called `ResolveSubfolder` unqualified from within the test namespace; every other existing test file qualifies APLSource calls with `#.` (`#.CheckFolder`, `#.ScanFolder`, `#.BuildFileMeta`), and this one did not. The mistake surfaced as a confusing "works when fixed directly, `VALUE ERROR: Undefined name` when run via the full suite" symptom (unqualified lookup from inside a nested test namespace does not search `#`) - fixed before RED was captured, so the RED evidence below is against the corrected file.

### Anticipated implementation

Bind `realpath` via `⎕NA` against `libc.so.6`. Join `root`/`relFolder` with absolute-wins semantics. Split the joined path into segments and try `realpath` on progressively shorter prefixes until one resolves. Check the resolved prefix is `root` or `root,'/'`-prefixed. Re-append whatever trailing segments were backed off.

## Cycle 2: GREEN

### Implementation

`apl-service/APLSource/ResolveSubfolder.aplf` as anticipated. `RealPath` is bound once per process (guarded by `⎕NC`) rather than on every call, since `⎕NA` registration is a workspace-level side effect. `root` itself is canonicalised once at the top of the function (falling back to `root` unchanged if that fails) so a root reached via a symlink still compares consistently against resolved candidates, even though the acceptance criteria do not test this specific case.

### How it was tested

`dyalogscript apl-service/tests/run-tests.apls` passes 62/62 with exit code 0: the eight new tests plus the 54 pre-existing ones, no regressions. The algorithm was additionally validated directly (outside the committed test file, via a throwaway script against the same fixture tree) against all nine scenarios - including the two not covered by named acceptance criteria (a nested valid subfolder, and a not-yet-existing subfolder) - before the final implementation and test files were written, to avoid repeating the kind of fixture-construction mistake found in issue #10. The repo configures no APL linter or formatter.
