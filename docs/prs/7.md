# PR notes: Issue #7 - LoadAppConfig

Plan: https://github.com/mkromberg/media-browser/blob/main/docs/plans/original-plan.md#L135

Part of epic #1 (Bootstrap and root listing). Follows #2 (vendored Jarvis and scaffolded config).

## Scope

`LoadAppConfig path` reads and parses an app config file, validates a non-empty `rootFolder`, and returns `(0 cfg)` or `(1 message)`, never signalling. `path` is always an absolute path.

`Initialize` and its config-path resolution are deferred to the epic-1 issue that stands up the end-to-end harness (#10). A running Dyalog interpreter can neither set an environment variable (`SetEnvironment` signals `DOMAIN ERROR`) nor change its working directory, so `Initialize`'s selection of a config file cannot be exercised in-process; the e2e harness, which launches a real server process with `$APL_SERVICE_APP_CONFIG` pointed at a fixture, is where that behaviour is both implemented and tested. The config-file path is defined to be absolute (see the plan): `$APL_SERVICE_APP_CONFIG` when set, otherwise an absolute path derived from the service's install location, never resolved against the process working directory. `LoadAppConfig` is the pure, path-in half of that design and is delivered here on its own.

## Cycle 1: RED

### Test surface

`apl-service/tests/unit/LoadAppConfigTests.aplf` (6 `Test_*` functions) drives `LoadAppConfig` directly and purely, in-process, against four JSON fixtures under `apl-service/tests/fixtures/config/` (`valid.json`, `malformed.json`, `missing-key.json`, `empty-root.json`) plus two paths that do not name a readable file (a non-existent file and the fixtures directory itself).

`ReturnsErrorMessage` is a shared predicate: a failure result is return code 1 and a non-empty character message. Reaching the predicate at all proves `LoadAppConfig` returned rather than signalled, since a signal would abort the test first; this is how "never signals" is asserted across every failure case.

### Anticipated implementation

`APLSource/LoadAppConfig.aplf`: `⎕NGET` the path under a trap (reporting a read error), `⎕JSON` under a trap (reporting a parse error), then validate `rootFolder` is present and non-empty; return `(0 cfg)` or `(1 message)`.
