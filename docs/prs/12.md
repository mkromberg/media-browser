# PR notes: Issue #12 - folder query parameter and error-code mapping in Get

Plan: https://github.com/mkromberg/media-browser/blob/main/docs/plans/original-plan.md#L115-L123

Part of epic #3.

## Scope

`GET /files?folder=<relative-path>` retargets the existing `ScanFolder`/`BuildFileMeta` pipeline at `ResolveSubfolder AppState.RootFolder folder`, so the same listing endpoint browses into any subfolder of the root, not just the root itself. `Get` maps `ResolveSubfolder`/`CheckFolder` outcomes to HTTP status:

| Outcome | HTTP response |
| --- | --- |
| `ResolveSubfolder` returns `'outside-root'` | 400 |
| `CheckFolder` returns `'not-found'` | 404, whether for the root or a subfolder |
| `CheckFolder` returns `'not-a-directory'` | 500 for the root itself, 404 for a requested subfolder |
| `CheckFolder` returns `'ok'`, then an error surfaces while listing (for instance an unreadable folder) | 500, via the existing `:Trap 0` |

The root/subfolder distinction for `'not-a-directory'` is decided by comparing the resolved path against `AppState.RootFolder` directly: `ResolveSubfolder` returns `root` unchanged for an empty `folder` parameter, so an unmodified root request resolves to exactly `AppState.RootFolder`.

## Behavioural change from issue #10's `Get`

Issue #10's `Get` called `BuildListingResponse AppState.RootFolder` directly, with no `CheckFolder` step; a root folder that did not exist failed inside `BuildListingResponse` and was caught by `:Trap 0` as a 500. Routing every request through `CheckFolder` now, this issue's mapping table applies uniformly to the root and to subfolders alike, so a missing root folder now returns 404 rather than 500. A root folder that exists but is not a directory, or exists but is unreadable, still reaches 500 exactly as before, just via `CheckFolder`'s `'not-a-directory'` outcome or the `:Trap 0` fallback respectively rather than an unconditional catch-all.

### Anticipated implementation

`Get` reads the `folder` query parameter from `req.QueryParams` (defaulting to `''` when absent), resolves it with `AppState.RootFolder ResolveSubfolder folder`, and on `'outside-root'` fails 400 immediately. Otherwise `CheckFolder` the resolved path: `'not-found'` fails 404; `'not-a-directory'` fails 500 if the resolved path is the root itself (`resolved≡AppState.RootFolder`) or 404 otherwise; anything else falls through to the existing `:Trap 0`-wrapped `BuildListingResponse` call.

## Cycle 2: GREEN

### Implementation

`apl-service/APLSource/Get.aplf` as anticipated. The `folder` lookup is `(req.QueryParams[;1]⍳⊂'folder')⊃req.QueryParams[;2],⊂''`: an index-and-pick against the two-column matrix, defaulting to `''` when `'folder'` is absent (index past the end picks the appended `⊂''`).

### Empirical finding: stub construction and Jarvis's real `QueryParams` cell depth do not match by default

`req.QueryParams[;1]⍳⊂'folder'` failed to find a match against every `StubReqFolder`-built request, even though both sides looked identical printed. Building the matrix as a struck-together literal, `1 2⍴(⊂'folder')(⊂folder)`, gives each cell a depth of 3, not 2: stranding two operands that are themselves already-enclosed scalars encloses each one a further level, on top of the enclosure already present, rather than leaving it as-is. Confirmed against a real `Jarvis.Request` instance (constructed directly via `⎕NEW Jarvis.Request` with a query string, bypassing the network layer) that its own `QueryParams` cells are depth 2 - matching a bare `⊂'folder'`, not the doubly-enclosed version. `StubReqFolder` now builds the matrix with indexed assignment (`params[1;1]←⊂'folder'`), which does not re-enclose and produces cells at the correct depth, matching Jarvis's own shape exactly.
