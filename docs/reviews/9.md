# Review: Issue #9 - BuildFileMeta and BuildListingResponse

## Summary

The implementation matches the plan and issue scope: `BuildFileMeta` builds the seven documented fields per file (no EXIF, as scoped), and `BuildListingResponse` orchestrates `ScanFolder` + `BuildFileMeta` into `.folder .files .subfolders`. The acceptance criterion, the `⎕JSON` array-shape invariant at 0/1/N matches, is met and directly tested using the same `1 ⎕JSON⍠'HighRank' 'Split'` incantation Jarvis itself uses. Independent verification with `dyalogscript` (percent-encoding of UTF-8 bytes, `⎕NPARTS` on dotfiles/no-extension/space/non-ASCII paths, the `0∊⍴` shape-based filter, and dyadic-each pairing at 0 and 1 elements) confirms every edge case the tests rely on behaves as the tests assume. All 47 tests pass, no regressions. One minor duplication and one minor portability note below; neither blocks merge.

## Findings

### Minor: extension normalisation is duplicated between BuildFileMeta and ClassifyExtension

`BuildFileMeta` strips a leading dot and lower-cases the extension (lines 12-13) before passing the already-normalised value to `ClassifyExtension`, which performs the identical strip-and-lower-case step internally. The result is correct either way (the second pass is a no-op on already-normalised input), but the normalisation logic now exists in two places. Passing the raw `fext` (with dot, original case) straight to `ClassifyExtension` would give the same classification and let `BuildFileMeta` keep its own copy solely for the `.extension` field, removing the duplication without changing behaviour.

Location: `apl-service/APLSource/BuildFileMeta.aplf:12-14`

### Note: `BuildFileMeta`'s percent-encoder assumes `⎕IO=1`

`hex[1+16 16⊤⍵]` (in the `Enc` helper) relies on 1-origin indexing. `ScanFolder`'s header comment calls out ⎕IO-independence explicitly (via destructuring rather than indexed pick) because Jarvis sets `⎕IO←1` workspace-wide at load (`vendor/jarvis/Jarvis.dyalog:5`) and the default interpreter `⎕IO` is also 1, so this is not a live bug in this codebase. `CheckFolder`, `LoadAppConfig`, and `FormatTimestamp` make the same implicit assumption, so this is consistent with existing practice, not a new pattern. Flagged only as a note since ⎕IO-independence is an explicit design goal called out for `ScanFolder` specifically.

Location: `apl-service/APLSource/BuildFileMeta.aplf:22`
